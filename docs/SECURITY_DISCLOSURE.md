# VRP Security Disclosure Policy

## Purpose

This document defines the public security disclosure policy for the VRP pilot.

The policy explains how security observations, vulnerabilities, implementation concerns, and responsible disclosure should be handled during the public pilot program.

The objective is to improve the security of the pilot while protecting participants and preserving the confidentiality of the protected VRP runtime.

---

## Scope

This policy applies to:

- public pilot documentation;
- public repositories;
- deployment guidance;
- validation materials;
- public interfaces;
- exported evidence.

This policy does not apply to:

- confidential implementation details;
- protected runtime source code;
- internal engineering processes;
- private pilot engagements.

---

## Disclosure Principles

Security reports should be:

- responsible;
- reproducible;
- technically accurate;
- supported by evidence.

Reports based on speculation alone should not be treated as confirmed findings.

---

## Responsible Disclosure

Individuals discovering a potential issue are encouraged to:

- document the observation;
- provide reproducible steps where possible;
- include relevant evidence;
- avoid unnecessary disclosure of confidential information.

The objective is coordinated remediation rather than public exposure before evaluation.

---

## Reporting Expectations

A useful report should include, when available:

- affected public component;
- environment description;
- observed behavior;
- expected behavior;
- reproduction steps;
- supporting evidence;
- impact assessment.

---

## Out of Scope

The following are generally outside the scope of the public pilot disclosure process:

- requests for protected runtime source code;
- proprietary implementation details;
- confidential engineering documentation;
- speculative implementation analysis without observable evidence.

---

## Public Communication

Verified security findings may result in:

- documentation updates;
- clarification of deployment guidance;
- compatibility notes;
- release notes;
- public advisories where appropriate.

Confidential implementation details will not be disclosed through public advisories.

---

## Coordinated Resolution

Where practical, identified issues should be:

1. reviewed;
2. reproduced;
3. evaluated;
4. documented;
5. addressed;
6. publicly communicated if appropriate.

---

## Security Boundary

The disclosure process intentionally protects:

- protected runtime implementation;
- proprietary algorithms;
- confidential engineering processes;
- internal authority logic;
- private operational procedures.

Public discussion should focus on observable behavior and documented interfaces.

---

## Disclosure Invariants

The security disclosure policy preserves the following principles.

- responsible disclosure is encouraged;
- observable evidence supports reported findings;
- protected implementation remains confidential;
- public documentation remains accurate;
- security communication remains technically grounded.

---

## Summary

The VRP security disclosure policy encourages responsible reporting of publicly observable issues while maintaining the confidentiality of the protected VRP runtime and supporting continuous improvement of the public pilot.
